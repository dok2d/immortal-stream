# immortal-stream

Отказоустойчивый стриминговый релей с 3-слойным композитингом. Поддерживает постоянное соединение с целевыми сервисами (YouTube, Twitch, Restream и др.) — даже когда входящий поток отсутствует, нестабилен или имеет неизвестный формат.

---

## Как это работает

```
 Основной поток  ─┐
 Резервный поток ─┤  RTMP / RTSP / SRT / HLS
 Аварийный поток ─┘
        │
        ▼
  ┌─────────────┐  опрос API  ┌──────────────────────┐
  │  mediamtx   │────────────▶  Python-оркестратор    │
  │  (приём)    │             └──────────┬───────────┘
  └─────────────┘                        │ выбор по приоритету
                                         │ + автопереключение
                                         ▼
                              ┌────────────────────────┐
  Заглушка     ──────────────▶ │   Композитор FFmpeg    │
  Оверлей      ──────────────▶ │   (3-слойный композит) │
                              └──────────┬─────────────┘
                                         │ UDP/MPEG-TS
                                         ▼
                              ┌────────────────────────┐
                              │  Выходной FFmpeg        │  ──▶  YouTube
                              │  (не перезапускается)   │  ──▶  Twitch
                              └────────────────────────┘  ──▶  …
```

### Слои

| Приоритет | Слой | Видимость |
|-----------|------|-----------|
| Базовый | **Заглушка (Placeholder)** | Всегда отправляется на целевой сервис. Отображается когда нет входящего потока. Поддерживает трансляцию. Варианты: чёрный экран, тестовая таблица (цветные полосы + часы), изображение или зацикленное видео. Поверх любого базового типа можно наложить текстовую надпись. |
| Средний | **Входящий поток** | Принимается любой протокол и кодек (RTMP, RTSP, SRT, HLS). Заменяет заглушку пока активен. |
| Верхний | **Оверлей** | Изображение или текст поверх входящего потока. Виден **только** когда поток активен. |

### Гарантия непрерывного вещания

Выходной процесс FFmpeg подключается к целевому сервису один раз при запуске и **никогда не разрывает соединение**. Композитор отправляет закодированное видео выходному процессу по безсоединительному каналу UDP/MPEG-TS через loopback-интерфейс. Кратковременные перезапуски композитора (~1 с) при переключении потоков вызывают короткую паузу в UDP-потоке, но выходной FFmpeg сохраняет RTMP-соединение и возобновляет передачу сразу после возвращения пакетов.

### Резервные источники

Можно настроить несколько входных источников с приоритетами. Композитор всегда использует активный поток с наивысшим приоритетом. При его отключении система мгновенно переключается на следующий доступный источник — RTMP-соединение с YouTube/Twitch не прерывается. Когда источник с более высоким приоритетом возвращается, он немедленно занимает место текущего.

## Быстрый старт

### 1. Сборка образа

```sh
podman build -f Containerfile -t immortal-stream .
```

Для ARM-хостов (Raspberry Pi 4, Apple Silicon VM и т.д.):

```sh
podman build -f Containerfile --build-arg TARGETARCH=arm64 -t immortal-stream .
```

### 2. Конфигурация

```sh
cp config.example.yaml config.yaml
$EDITOR config.yaml
```

Медиафайлы (заглушка, оверлей, шрифты) монтируются в `/media` внутри контейнера.

### 3. Запуск

```sh
podman run -d --name immortal-stream \
  --restart unless-stopped \
  -v "$(pwd)/config.yaml:/etc/immortal-stream/config.yaml:ro" \
  -v "$(pwd)/media:/media:ro" \
  -p 1935:1935/tcp \
  -p 8890:8890/udp \
  immortal-stream
```

Или через Compose:

```sh
podman-compose up -d
# или
docker compose up -d
```

| Порт | Протокол | Назначение |
|------|----------|------------|
| 1935 | TCP | Приём RTMP |
| 8890 | UDP | Приём SRT |
| 8888 | TCP | Приём HLS (опционально, выключен по умолчанию) |

## Справочник по конфигурации

Все параметры — в одном YAML-файле. Полностью аннотированный пример: [`config.example.yaml`](config.example.yaml).

### `log_level`

```yaml
log_level: INFO   # DEBUG | INFO | WARNING | ERROR
```

Также можно задать через переменную окружения `LOG_LEVEL`; значение из конфига имеет приоритет.

### `ingest`

```yaml
ingest:
  port: 1935               # Порт приёма RTMP
  srt_port: 8890           # Порт приёма SRT
  hls: false               # Включить приём HLS (порт 8888)
  hls_port: 8888           # Порт приёма HLS
  stream_key_required: false
  allowed_key: ""          # используется только при stream_key_required: true
```

При `stream_key_required: true` принимаются только потоки, опубликованные на `rtmp://host:1935/live/<allowed_key>`. Потоки на других путях игнорируются.

### `ingest.redundant_sources` — резервные источники с автопереключением

```yaml
ingest:
  redundant_sources:
    - primary    # rtmp://host:1935/live/primary   (наивысший приоритет)
    - backup     # rtmp://host:1935/live/backup
    - emergency  # rtmp://host:1935/live/emergency (наинизший приоритет)
```

При заданном `redundant_sources` оркестратор отслеживает все перечисленные источники одновременно и всегда подаёт на композитор **подключённый источник с наивысшим приоритетом**:

- Все источники могут быть подключены одновременно. Источники с низким приоритетом находятся в режиме ожидания и не потребляют ресурсы композитора.
- При отключении активного источника система **мгновенно переключается** на следующий доступный — выходной FFmpeg не перезапускается, RTMP-соединение с YouTube/Twitch не прерывается.
- Когда источник с более высоким приоритетом переподключается, он **немедленно возвращается** в композитор.

Telegram-уведомления сообщают о каждом подключении/отключении резервного источника, каждом вытеснении и переключении с указанием IP, протокола и кодека.

Оставьте `redundant_sources` пустым (по умолчанию), чтобы принимать любой одиночный поток на `/live/*` (устаревший режим «кто первый подключился»).

### `placeholder`

```yaml
placeholder:
  type: testcard           # black | testcard | image | video
  # Текст поверх любого базового типа (опционально):
  # text: "Трансляция скоро начнётся"
  # font_path: /media/fonts/MyFont.ttf   # TTF/OTF; JetBrains Mono если не указан
  # font_size: 72
  # font_color: white                    # Имя цвета FFmpeg или #RRGGBB
  # Заглушка-изображение/видео:
  # path: /media/holder.jpg  # обязательно для image/video
  x: 0                      # позиция (text: 0,0 = по центру)
  y: 0
  opacity: 1.0
```

Заглушка перекодируется в заданное выходное разрешение. `testcard` показывает цветные полосы с часами. Изображения дополняются чёрными полосами для сохранения пропорций. Видео зацикливается. Опциональное поле `text` рисует надпись поверх любого базового типа.

### `overlay`

```yaml
overlay:
  enabled: true
  type: image              # image | text
  path: /media/logo.png    # для type: image (PNG/JPEG)
  # text: "LIVE"           # для type: text
  # font_path: /media/font.ttf
  # font_size: 48
  # font_color: white
  x: 20
  y: 20
  opacity: 0.9
```

Оверлей накладывается **только** пока входящий поток активен.

### `output`

```yaml
output:
  targets:
    - rtmp://a.rtmp.youtube.com/live2/YOUR_KEY
    - rtmp://live.twitch.tv/app/YOUR_KEY
  video:
    width: 1920
    height: 1080
    fps: 30
    bitrate: 6000k
    preset: ultrafast      # x264-пресет; ultrafast = минимальная нагрузка на CPU
    tune: zerolatency      # x264 tune (zerolatency|film|animation|grain|...)
    gop: 60                # должен быть >= fps (минимум 1 с между ключевыми кадрами)
  audio:
    bitrate: 128k
    sample_rate: 44100
```

Несколько целей отправляются одновременно через tee-мультиплексор FFmpeg. Все используют копирование кодека — перекодирование происходит только в композиторе, а не для каждой цели отдельно.

### `telegram`

```yaml
telegram:
  enabled: true
  bot_token: "123456:ABC..."
  chat_id: "-1001234567890"
```

Уведомления: поток начался (с указанием IP / протокола / кодека / разрешения / FPS), поток остановился (с длительностью), переключение на резерв, вытеснение по приоритету, перезапуски процессов, ошибки.

Команды бота для управления в реальном времени:

| Команда | Описание |
|---------|----------|
| `/menu` | Главное меню с inline-кнопками |
| `/status` | Текущее состояние потока и настройки |
| `/placeholder black\|testcard\|text\|image\|video\|opacity\|pos` | Изменить заглушку |
| `/overlay off\|text\|image\|x\|y\|opacity\|size\|color` | Настроить оверлей |
| `/target list\|add\|remove\|set` | Управление RTMP-целями |
| `/output bitrate\|fps\|size\|preset` | Изменить параметры кодирования |
| `/stop` | Приостановить все процессы (композитор + выход) |
| `/resume` | Возобновить все процессы |
| `/help` | Показать доступные команды |

Бот также принимает медиафайлы: отправьте фото для установки заглушки/оверлея, или видео для заглушки-видео. Изменения применяются немедленно, но **не сохраняются** — перезапуск контейнера возвращает настройки из конфиг-файла.

## Источники входящего потока

Подключите энкодер или источник:

| Протокол | Формат URL |
|----------|-----------|
| RTMP | `rtmp://host:1935/live` или `rtmp://host:1935/live/<ключ>` |
| SRT | `srt://host:8890` |
| HLS | `http://host:8888/live` (при включении) |

Принимается любой кодек и разрешение; композитор перекодирует в заданные выходные параметры. Аудио опционально — при отсутствии аудиодорожки генерируется тишина.

## Безопасность

- Контейнер работает от **непривилегированного пользователя** (uid 1000).
- Внутренний RTSP-путь между mediamtx и композитором защищён случайно сгенерированным токеном, обновляемым при каждом запуске контейнера.
- Конфигурация и медиафайлы монтируются **только для чтения**.
- Наружу проброшены только порты приёма. Все внутренние компоненты общаются через loopback-интерфейс.
- Исходящие соединения только к настроенным RTMP-целям и Telegram API (если включён).
- Усиление безопасности в compose.yaml: `no-new-privileges`, `cap_drop: ALL`, `tmpfs` для временных файлов.
- Встроенный healthcheck опрашивает API mediamtx для проверки работоспособности сервиса.

## Решение проблем

**Поток принимается, но на выходе чёрный экран / зависшая картинка**
- Проверьте правильность пути к файлу заглушки и его доступность внутри контейнера.
- Запустите с `LOG_LEVEL=DEBUG` для просмотра команд FFmpeg.

**Публикация отклонена / поток не обнаружен**
- Установлен `stream_key_required: true` — подключайтесь к `rtmp://host:1935/live/<allowed_key>`.
- При использовании `redundant_sources` подключайтесь к `rtmp://host:1935/live/<имя_источника>`.

**Выходной FFmpeg сразу завершается**
- Убедитесь, что URL в `output.targets` доступны из контейнера.
- Проверьте валидность ключа трансляции на целевом сервисе.

**Высокая нагрузка на CPU**
- Установите `preset: ultrafast` (по умолчанию). При необходимости уменьшите разрешение или частоту кадров.
- Убедитесь, что видео-заглушка уже в нужном разрешении, чтобы избежать затратного масштабирования.

**Просмотр логов**
```sh
podman logs -f immortal-stream
```
